CURSOR PROMPT — Retrieval Middleware Proof-of-Concept
======================================================

Build a retrieval middleware proof-of-concept in Python. The goal is a lightweight middleware layer that sits between a user query and a vector database, with a semantic cache and a re-ranking step. Here is the full spec:


---
PROJECT STRUCTURE
---

retrieval_middleware/
├── main.py                  # FastAPI app + endpoints
├── semantic_cache.py        # Semantic cache logic
├── reranker.py              # Cross-encoder re-ranking logic

---
COMPONENT 1: Vector DB Client (vector_db_client.py)
---

Create a mock vector DB client that simulates Pinecone/Milvus/Weaviate behavior. It should:
- Store a set of 200+ hardcoded or randomly generated "documents" (each with an id, text, and embedding vector)
- Expose a query(embedding, top_k=50) method that computes cosine similarity between the query embedding and all stored document embeddings, then returns the top_k results with scores
- Add a simulated network latency of 80–150ms using asyncio.sleep with a random value in that range, to realistically mimic a real vector DB round trip
- Use numpy for embedding generation and similarity computation
- Generate document embeddings once at startup using random vectors of dimension 384


---
COMPONENT 2: Semantic Cache (semantic_cache.py)
---

Build an in-memory semantic cache. It should:
- Store recently seen queries as (embedding, results) pairs in a list or dict structure
- On each new query, compute cosine similarity between the new query's embedding and all cached query embeddings
- If the highest similarity score exceeds a configurable threshold (default: 0.92), return the cached results as a cache hit
- If no match, execute the full pipeline and store the result in the cache
- Support a max cache size (default: 100 entries); evict the oldest entry when full (FIFO)
- Expose methods: get(query_embedding) → returns (hit: bool, results, similarity_score) and set(query_embedding, results)
- Track hit/miss counts for stats


---
COMPONENT 3: Re-ranker (reranker.py)
---

Build a re-ranker using a Cross-Encoder model. It should:
- Load the cross-encoder/ms-marco-MiniLM-L-6-v2 model from sentence-transformers at startup
- Expose a rerank(query: str, documents: list[dict], top_n: int = 5) method
- The method takes the original query string and the list of candidate documents returned by the vector DB (up to 50), scores each (query, document_text) pair using the cross-encoder, and returns the top_n documents sorted by cross-encoder score
- Include timing instrumentation: return how many milliseconds the re-ranking step took


---
COMPONENT 4: Embedding Service (inline in main.py or a helper)
---

Use sentence-transformers with the all-MiniLM-L6-v2 model to convert query strings into 384-dimension embeddings. This same model should be used for both the semantic cache similarity checks and for querying the mock vector DB.


---
COMPONENT 5: FastAPI App (main.py)
---

Build a FastAPI application with the following:

POST /query
- Request body: { "query": "string", "top_n": int (default 5) }
- Pipeline:
  1. Embed the query using the embedding model — record time
  2. Check semantic cache — record time
  3. If cache hit: skip vector DB call, skip re-ranking, return cached results with cache_hit: true
  4. If cache miss: call vector DB query() for top 50 results — record time
  5. Pass top 50 results to re-ranker, get top_n back — record time
  6. Store result in semantic cache
  7. Return results with cache_hit: false
- Response body must include:
  - results: list of top_n documents (id, text, score)
  - cache_hit: bool
  - cache_similarity_score: float (the best similarity found, even on a miss)
  - latency_breakdown: object with millisecond timings for each stage:
      - embedding_ms
      - cache_lookup_ms
      - vector_db_ms (0 if cache hit)
      - reranking_ms (0 if cache hit)
      - total_ms

GET /stats
- Returns cache hit count, miss count, hit rate percentage, and current cache size

DELETE /cache
- Clears the semantic cache and resets stats

COMPONENT 7: Tests (tests/)
---

Write pytest tests covering the following scenarios:

test_cache.py:
- Test that an identical query returns a cache hit with similarity score of 1.0
- Test that a semantically different query returns a cache miss
- Test that cache eviction works correctly when max size is exceeded
- Test the GET /stats endpoint reflects correct hit/miss counts

test_reranker.py:
- Test that reranker returns exactly top_n results
- Test that results are sorted by descending cross-encoder score
- Test that reranker handles edge case where fewer than top_n documents are passed in

test_middleware.py:
- Test cold query end-to-end: verify cache_hit=false, all latency fields are populated, results list has correct length
- Test warm query: send the same query twice, verify second call has cache_hit=true and vector_db_ms=0 and reranking_ms=0
- Test near-duplicate query: send a paraphrased version of a prior query and verify it returns a cache hit
  (use queries like "What is machine learning?" followed by "Define machine learning" — these should be
  close enough to exceed the 0.92 threshold)
- Test that /cache DELETE resets behavior so the next query is a cold miss again

RUNNING INSTRUCTIONS (add to README.md)
---

1. Install dependencies:
   pip install -r requirements.txt

2. Start the server:
   uvicorn main:app --reload

3. Explore the API interactively via Swagger UI:
   http://localhost:8000/docs

4. Run all tests:
   pytest tests/ -v

5. Sample curl — cold query:
   curl -X POST http://localhost:8000/query \
     -H "Content-Type: application/json" \
     -d '{"query": "What is machine learning?", "top_n": 5}'

6. Sample curl — warm query (repeat the same query to trigger cache hit):
   curl -X POST http://localhost:8000/query \
     -H "Content-Type: application/json" \
     -d '{"query": "What is machine learning?", "top_n": 5}'

7. Check cache stats:
   curl http://localhost:8000/stats

8. Clear the cache:
   curl -X DELETE http://localhost:8000/cache
